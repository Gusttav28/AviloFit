# Ask Avilo Aceternity Demo Card Design

## Design Goals

This design satisfies R1-R15 by replacing the current locally reproduced Ask Avilo gooey area with an Aceternity-registry-backed card that matches the uploaded reference structure: rounded white box, prompt text at top, and Gooey Input below in the same box.

The design intentionally authorizes the dependency path Gustavo requested, but keeps product behavior bounded to local UI only. The card is not an AI chat surface and must not send or persist the user's text.

## Current-System Observations

- `/dashboard` renders through `app/dashboard/page.tsx` and `components/dashboard/dashboard-screen.tsx`.
- `DashboardScreen` composes `.reference-grid`, `.dashboard-left-column`, `.summary-column`, the `GoalProgress` region, `AskAviloGooeyInput`, `.dashboard-right-column`, `.activity-column`, and history.
- `AskAviloGooeyInput` is currently exported from `components/dashboard/ask-avilo.tsx` and already renders below the progress cards.
- The current compact entry locally reproduces gooey behavior without package changes. This work supersedes that no-dependency choice because Gustavo explicitly requested trying the shadcn/Aceternity registry demo.
- `components/dashboard/ask-avilo.tsx` also contains the older provider-backed `AskAvilo` assistant. This task must not wire that full assistant or provider behavior into `/dashboard`.
- `app/globals.css` contains `.ask-avilo-gooey` styles, dashboard layout styles, responsive breakpoints, focus-visible styles, and reduced-motion handling.
- Existing tests under `tests/dashboard/dashboard-screen.test.tsx` and `e2e/dashboard.spec.ts` already cover Ask Avilo placement, local behavior, no requests, responsive geometry, and dashboard preservation. They must be updated for the new card copy, structure, registry behavior, and dependency decision.
- `package.json` currently has no `motion` dependency. This design permits adding it only if the Aceternity registry component installs or requires it.

### New Files Expected From Registry Install

The shadcn command may generate Aceternity component files under `components/ui/`. The expected allowed pattern is:

- `components/ui/gooey-input.tsx`
- `components/ui/gooey-input-demo.tsx`
- Other `components/ui/*gooey*input*.tsx` files directly generated by the Aceternity registry item.

If the registry generates support files outside `components/ui/`, the Implementer must inspect them. They may remain only when required by the generated Gooey Input component and within established project conventions. Otherwise, stop for spec revision.

### New Manual Files

No new manual application files are expected. A single focused dashboard test file under `tests/dashboard/` is allowed only if keeping all tests in `dashboard-screen.test.tsx` would make the test file unclear. No routes, APIs, fixtures, providers, models, migrations, auth, telemetry, config, or unrelated component files may be created.

## Install And Fallback Design

The Implementer must use this exact sequence after R1 is satisfied:

1. Run `npx.cmd shadcn@latest add @aceternity/gooey-input-demo`.
2. Inspect output and `git status --short`.
3. If the demo item is unavailable, not found, or generates no usable component, run `npx.cmd shadcn@latest add @aceternity/gooey-input`.
4. Inspect generated files and package changes before editing integration code.
5. If both commands fail or produce incompatible/uninspectable code, stop for human decision.

The fallback to `@aceternity/gooey-input` is preferred over stopping immediately because the official Aceternity page documents that command and the catalog identifies `gooey-input` as the canonical component. Stopping after both attempts protects the work from silently drifting back into a hand-rolled static input.

## Component Structure

`AskAviloGooeyInput` should render one card-like region with the prompt text and the Aceternity input below it:

```text
section.ask-avilo-card[aria-labelledby]
|- p or heading.ask-avilo-card-copy
|  `- Ask anything about your recipes or workouts to Avilo AI
`- div.ask-avilo-card-control
   `- Aceternity Gooey Input integration
```

Implementation guidance:

- Keep the card in `components/dashboard/ask-avilo.tsx` because this file already owns both the existing compact dashboard entry and older assistant surface.
- Import or inline the generated `components/ui/*gooey*input*` component only after inspecting generated source.
- Prefer using the generated component through a small dashboard wrapper rather than copying the entire demo into the dashboard file.
- Keep the top copy outside the generated input so the same card structure matches the reference image.
- If the generated component includes its own example text `How is your business management going?`, replace it in the dashboard integration with `Ask anything about your recipes or workouts to Avilo AI` or remove the duplicate internal example so the old business copy never renders.
- If the generated component does not expose props for placeholder/label/value/state, adapt the generated component minimally under `components/ui/` to support the dashboard wrapper while preserving Aceternity behavior.

## Card Visual Design

The Ask Avilo card should follow the uploaded reference image:

- White or near-white card surface.
- Rounded rectangle, not a separate floating pill alone.
- Compact prompt text at the top-left inside the card.
- Input/search region below the text within the same card.
- A close/clear/search affordance may sit at the right side if provided by Aceternity behavior.
- Soft border or shadow consistent with the existing dashboard, without becoming a hero or marketing card.
- Stable padding and dimensions so text, input, and icon controls do not overlap.

Avoid:

- Nested cards.
- Decorative blobs, orbs, or unrelated background effects.
- A huge hero-sized AI prompt.
- Moving the card outside the left dashboard column.

## Layout And Placement

`DashboardScreen` must preserve this order:

```text
.reference-grid
|- .dashboard-left-column
|  |- .summary-column
|  |- Progress goals region / .progress-row
|  `- AskAviloGooeyInput / .ask-avilo-card
`- .dashboard-right-column
   |- .activity-column
   `- history region
```

Placement requirements:

- The card is a normal-flow sibling after the progress region.
- The card is not a descendant of `.summary-column`, `.progress-row`, `.progress-card`, `.activity-column`, or history.
- At desktop widths, align card x/width to the progress region or the left column within a small tolerance.
- At stacked widths, the card follows the progress cards before the right-column content.
- Natural vertical height from the card is acceptable; unrelated dashboard layout changes are not.

## Behavior Design

The installed Aceternity behavior is the behavioral source of truth after generation. The dashboard wrapper must preserve:

- Focus/click open behavior.
- Local typing.
- Gooey visual transition or filter/motion merge.
- Clear/dismiss/search affordance behavior present in the generated component.
- Escape/blur/Enter recovery behavior where present or where needed to keep interactions local and safe.
- Reduced-motion safety.

The dashboard wrapper must add or enforce:

- No form submission that sends data.
- No fetch/provider/router/storage/telemetry side effect.
- No response/status panel.
- Local-only value state if a controlled input is required.
- Clear accessible names for the input and any icon button.

If preserving exact Aceternity behavior conflicts with preventing external side effects, the no-request/no-persistence boundary wins and the Implementer must document the adaptation.

## Accessibility Design

- The card region should have an accessible name from the visible top prompt text.
- The input/search element should have an accessible name related to `Ask anything about your recipes or workouts to Avilo AI`.
- Icon-only controls must have accessible names matching their real behavior, such as `Clear Ask Avilo input`.
- Focus-visible outlines must be visible on the input and controls and not clipped by the card or gooey filter.
- Placeholder text must not be the only accessible name.
- Keyboard order should reach the Ask Avilo input after earlier dashboard controls and before the clear/dismiss control inside the card, then continue normally.
- Axe serious/critical violations caused by the Ask Avilo card must be zero.

## Responsive Design

- `>= 1024px`: Card aligns with the left-column/progress width. Prompt and input remain within the same white card and avoid crowding the progress cards.
- `768px-1023px`: Card follows the stacked left column width. Input remains usable and visually below the prompt.
- `< 768px`: Card fits within viewport padding. Text may wrap naturally. Input text may scroll or truncate inside the input, but the page must not overflow horizontally. Touch targets should be at least 40px by 40px where practical.
- The card must preserve `min-width: 0` behavior wherever needed to prevent overflow.

## Reduced Motion And Performance

- The generated Aceternity component may use `motion`.
- Motion must be limited to the input/card interaction and must not animate unrelated dashboard layout.
- Respect `prefers-reduced-motion: reduce` by disabling or minimizing non-essential transitions/animations while preserving focus, typing, and clear/dismiss behavior.
- No continuous animation, flashing, JavaScript polling, canvas loop, or heavy background effect is authorized.

## Security, Privacy, And Health-Data Design

- User typed text must remain local to the component and vanish on reload/remount unless the generated component has an internal state lifecycle that clears similarly.
- Do not pass text to `AskAviloProvider`, APIs, server actions, routes, local/session storage, query strings, console logs, analytics, telemetry, fixtures, or model/provider code.
- Do not add health-data processing, consent flow, auth flow, medical advice, coaching advice, or diagnosis copy.
- The card may say Avilo AI in the prompt, but it must not imply a live response exists in this task.

## Dependency Impact

Authorized:

- Networked shadcn command sequence in Install And Fallback Design.
- `motion` dependency if installed/required by the Aceternity registry component.
- `package.json` and `package-lock.json` updates directly resulting from the registry dependency.

Not authorized:

- `framer-motion` unless the registry explicitly uses that exact dependency instead of `motion` and the Implementer stops to document why the official catalog expectation differed.
- Any unrelated packages.
- Script/config/package-manager changes.
- Manual dependency additions unrelated to the generated Aceternity component.

## Verification Design

Component tests should verify:

- The required recipe/workout Avilo AI copy appears once.
- `How is your business management going?` is absent.
- The prompt text and input/search control share one card container.
- The card is below the progress region and outside progress cards.
- The input/control has accessible role/name.
- Local typing, Enter, Escape, and clear/dismiss behavior cause no response/status/alert.
- Existing progress content is preserved.

E2E tests should verify:

- Viewports: 1440, 1024, 768, and 360.
- Card x/width align with progress region or left column.
- Card y is below progress region bottom.
- Prompt text is above input/search control inside the same card.
- Focus/click/typing/open/clear/Escape/blur interactions preserve Aceternity behavior and remain local.
- Request capture is reset after page load and remains empty for external or `/api` requests caused by the Ask Avilo card.
- No horizontal overflow.
- Keyboard path reaches input and controls with visible focus.
- Axe serious/critical violations scoped to the card are zero.
- Screenshots capture resting and interacted states where practical.

Required command checks:

- `npm.cmd run lint`
- `npm.cmd run typecheck`
- `npm.cmd run test`
- `npm.cmd run test:e2e` or a narrower documented Playwright command with justification if the full suite is impractical

Required review checks:

- `git status --short` before and after install.
- `git diff -- package.json package-lock.json` if package files changed.
- Changed-file review against R15.
- Generated component source review against R8 and R14.

## Alternatives Considered

| Alternative | Decision | Reason |
| --- | --- | --- |
| Try only the user-provided `@aceternity/gooey-input-demo` and stop if unavailable | Rejected | The official Aceternity page documents `@aceternity/gooey-input`, so a canonical fallback is safer and more likely to satisfy the behavior request. |
| Skip registry install and keep the current local reproduction | Rejected | Gustavo explicitly asked to try the shadcn/Aceternity demo and make sure it has the same behavior as the Aceternity component. |
| Install the documented component first | Rejected | Gustavo's exact demo command must be preserved as the primary attempt. |
| Build backend Ask Avilo chat behavior while touching the card | Rejected | Provider/API/persistence/health-data flows are explicitly out of scope. |
| Put the prompt and input in separate surfaces | Rejected | The reference image and user request require top text and input/search area inside the same rounded white box. |

## Stop Conditions

The Implementer must stop for human decision or spec revision before proceeding if:

- The governing Notion item is not `In Progress`.
- The checkout is not `feature/dashboard-experience`.
- Both registry commands fail or generate no usable Aceternity Gooey Input.
- The registry generates unexpected required files outside the allowed scope.
- Preserving Aceternity behavior appears to require backend/API/provider/persistence/telemetry/auth/health-data changes.
- Exact behavior cannot be preserved inside the rounded card without changing unrelated dashboard layout.
- Package changes include unrelated dependencies or config/script modifications.
- Implementation needs to edit files outside R15.
